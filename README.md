# FinalDemoExam — система учета товаров и заказов книжной компании

## Назначение проекта

WPF-приложение `FinalDemoExam` разработано для демонстрационного экзамена по предметной области книжной компании. Система помогает контролировать остатки товаров на складе, просматривать каталог, работать с заказами и разграничивать доступ пользователей по ролям.

## Что требуется по документации

### Модуль 1 — база данных и импорт

По заданию необходимо:

1. Проанализировать предметную область продажи книг.
2. Создать базу данных в 3 нормальной форме.
3. Обеспечить ссылочную целостность первичными и внешними ключами.
4. Импортировать данные из файлов с пометкой `import`.
5. Подготовить скрипт создания/наполнения базы данных.
6. Сформировать ER-диаграмму в PDF.

В проекте база разделена на сущности: товары, категории, производители, поставщики, единицы измерения, пользователи, роли, заказы, статусы заказов, пункты выдачи и состав заказа. Такое разделение убирает дублирование справочников и соответствует 3 НФ.

### Модуль 2 — интерфейс приложения

По заданию необходимо:

1. Реализовать окно авторизации первым экраном приложения.
2. Дать возможность входа по логину/паролю из БД.
3. Дать возможность входа гостем.
4. Показать ФИО пользователя в интерфейсе.
5. Реализовать роли:
   - гость — просмотр товаров без поиска, сортировки и фильтрации;
   - клиент — просмотр товаров без поиска, сортировки и фильтрации;
   - менеджер — просмотр товаров с поиском, сортировкой, фильтрацией и просмотром заказов;
   - администратор — полный доступ к товарам и заказам.
6. Вывести список товаров по шаблону из задания.
7. Подсветить товары по условиям скидки и отсутствия на складе.
8. Подключить логотип, иконку и единый стиль из руководства.
9. Подготовить блок-схему алгоритма разработки приложения в PDF.
10. Подготовить документ со скриншотами работоспособности.

### Модуль 3 — навигация и операции с данными

По заданию необходимо:

1. Организовать последовательную навигацию между окнами.
2. Реализовать добавление, редактирование и удаление товаров для администратора.
3. Реализовать просмотр заказов для менеджера и администратора.
4. Реализовать добавление, редактирование и удаление заказов для администратора.
5. Защитить действия по ролям.

## Функциональность приложения

### Авторизация

При запуске открывается окно `LoginWindow`. Пользователь вводит логин и пароль, после чего приложение ищет запись пользователя в базе данных вместе с ролью. При успешном входе сохраняются идентификатор, ФИО и роль пользователя, затем открывается главное окно. При ошибке показывается понятное сообщение. Также доступна кнопка входа гостем.

### Главное окно товаров

Окно `MainWindow` показывает список товаров из базы данных. Для каждого товара выводятся:

- изображение или заглушка `picture.png`;
- категория;
- наименование;
- описание;
- производитель;
- поставщик;
- цена;
- единица измерения;
- количество на складе;
- скидка.

Шаблон вывода товаров сохранен по заданию. Упрощено только отображение скидки: вместо лишнего `MultiBinding` используется готовое свойство `DiscountText`, поэтому XAML стал короче и понятнее.

### Поиск, сортировка и фильтрация

Для менеджера и администратора включается панель фильтров:

- поиск по артикулу, названию, описанию, категории, производителю и поставщику;
- сортировка по цене и остатку;
- фильтрация по диапазону скидки.

Для гостя и клиента эта панель скрыта, что соответствует заданию.

### Работа с товарами

Администратор может:

1. Нажать `Добавить товар`.
2. Заполнить карточку товара.
3. Выбрать категорию, поставщика и единицу измерения из справочников.
4. Указать производителя: если такого производителя еще нет, он создается автоматически.
5. Выбрать изображение товара.
6. Сохранить товар.
7. Открыть товар двойным щелчком для редактирования.
8. Удалить товар, если он не используется в заказах.

### Работа с заказами

Менеджер и администратор могут открыть список заказов. Администратор дополнительно может добавлять, редактировать и удалять заказы. В карточке заказа выбираются статус, даты, пункт выдачи, клиент и состав заказа.

## Руководство по стилю

По руководству используется:

- основной фон: `#FFFFFF`;
- дополнительный фон: `#7FFF00`;
- акцент целевого действия: `#00FA9A`;
- шрифт: `Times New Roman`;
- логотип и иконка из ресурсов проекта.

В текущей версии стили вынесены в общие ресурсы `App.xaml`: кнопки, поля ввода, выпадающие списки, даты, списки и окна имеют единые размеры, отступы, шрифт и границы. Это делает интерфейс более читаемым и убирает случайные расхождения между окнами.

## Как привести импорт к 3 НФ

Чтобы импорт из Excel был качественным, данные нужно не просто загрузить в одну таблицу, а разложить по справочникам и связям.

### 1. Очистить исходные файлы

Перед импортом нужно:

- удалить пустые строки и дубликаты;
- привести одинаковые названия к одному написанию;
- проверить числовые поля: цена, скидка, количество;
- проверить даты заказов и доставки;
- проверить, что у каждого товара есть артикул;
- проверить, что изображения из импорта есть в ресурсах.

### 2. Выделить справочники

Из файла товаров нужно выделить отдельные таблицы:

- `Categories` — категории товаров;
- `Manufacturers` — производители;
- `Suppliers` — поставщики;
- `Units` — единицы измерения.

Из файла пользователей:

- `Roles` — роли;
- `Users` — пользователи со ссылкой на роль.

Из файла заказов:

- `OrderStatuses` — статусы заказов;
- `PickupPoints` — пункты выдачи;
- `Orders` — заказы;
- `OrderItems` — товары в заказах.

### 3. Назначить ключи

Рекомендуемая структура ключей:

- `Products.article` — первичный ключ товара;
- `Categories.id`, `Manufacturers.id`, `Suppliers.id`, `Units.id` — первичные ключи справочников;
- `Users.id`, `Roles.id` — первичные ключи пользователей и ролей;
- `Orders.id` — первичный ключ заказа;
- `OrderItems.id` или составной ключ из заказа и товара — ключ состава заказа.

### 4. Настроить внешние ключи

Внешние ключи должны связывать:

- товар с категорией;
- товар с производителем;
- товар с поставщиком;
- товар с единицей измерения;
- пользователя с ролью;
- заказ с пользователем;
- заказ со статусом;
- заказ с пунктом выдачи;
- строку заказа с заказом и товаром.

### 5. Проверить 3 НФ

Таблица находится в 3 НФ, если:

1. Все поля атомарные — в одной ячейке одно значение.
2. Неключевые поля зависят от ключа полностью.
3. Нет транзитивных зависимостей: например, название категории не хранится в таблице товаров, а находится в таблице `Categories`.

Пример: плохо хранить в `Products` одновременно `category_id` и `category_name`. Правильно хранить только `category_id`, а название брать через связь с `Categories`.
